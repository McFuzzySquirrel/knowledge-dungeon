# Creator Phase

## The Goal

In Creator phase, you **design the dungeon** by creating topic-rooms and linking them. You're the architect of your knowledge structure.

## What You Do

1. **Add a root topic** — Start with a broad subject (e.g., "React")
2. **Create child topics** — Break it down (e.g., "Components", "Hooks", "State Management")
3. **Link them** — Define relationships: subtopic, prerequisite, analogy, depends_on, related
4. **Iterate** — Refine the structure until it feels complete
5. **Finish** — Mark phase complete when 90%+ of the map is designed

## The Dungeon Builder

In-game, the Creator UI lets you:

- Right-click rooms to create child rooms
- Drag-and-drop to position them visually
- Edit room names and descriptions
- Create cross-links between rooms
- Delete rooms (with confirmation)
- See a preview of the overall structure

## Edge Types Explained

| Type | Meaning | Example |
| --- | --- | --- |
| **subtopic** | A part of the whole | "Components" is a subtopic of "React" |
| **prerequisite** | Must learn this first | "Functions" prerequisite for "Closures" |
| **analogy** | Similar concept, different context | "Array.map()" analogous to "SQL SELECT" |
| **depends_on** | Technical dependency | "Error Handling" depends on "Try-Catch" |
| **related** | Nearby in learning | "Async" related to "Promises" |

## Philosophy

The Creator phase assumes you **understand the big picture**. You're externalizing your mental model into a visual structure. This act of design is itself learning.

## Design Insight

The Creator phase uses a **game-like interface** instead of a form. You walk around, click on rooms, visualize the structure. This makes designing a knowledge map feel like adventure, not a chore.
