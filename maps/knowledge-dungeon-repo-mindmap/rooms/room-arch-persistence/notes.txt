# Persistence Layer

## What It Does

The persistence layer is the **bridge between domain logic and storage**. It defines the types that get serialized to disk and the mechanisms for loading/saving.

## Core Types

### SubjectSnapshot
The complete serialized state of a dungeon:
```
{
  dungeon: DungeonMetadata,
  rooms: Record<string, RoomMetadata>
}
```

This is what gets written to `dungeon.json` or localStorage.

### DungeonMetadata
Contains:
- Subject name and ID
- Schema version
- Root room and all room/edge summaries
- Phase state
- Progression snapshot

### RoomMetadata
Contains per-room:
- Topic name
- State (Creator, Scribe, Archaeologist phases)
- Validation results
- Review pass count
- Attachment tracking

## Storage Backends

The app supports multiple storage backends via an abstraction:

### Web (localStorage)
- Stores subjects in browser localStorage
- Limited by storage quota (~5-10MB)
- Lost on cache clear (unless exported)

### Electron (filesystem)
- Stores subjects in `<userData>/dungeon-data/{subjectId}/dungeon.json`
- Automatic timestamped backups in `.backups/`
- Admin panel to export/import between machines

## Versioning

The schema version (`CURRENT_SCHEMA_VERSION = '1.0.0'`) allows:
- **Backward compatibility** — Upgrade old subjects without data loss
- **Migration hooks** — Transform data as schema evolves
- **Deprecation warnings** — Alert users to old formats

## Design Insight

Domain logic knows *nothing* about how data is stored. Only persistence types and services know about JSON, localStorage, or filesystem. This separation enables storage innovation without changing core logic.
